# RAG 系统架构对比

> 更新时间：2026-03-02
> 状态：主力系统已切换至 pgvector + bge-m3 混合检索

## 1. 系统对比总览

| 维度 | 旧系统 (ChromaDB) | 主力系统 (pgvector) | 提升点 |
|------|-------------------|---------------------|--------|
| **检索方式** | 纯向量 (Dense only) | 混合检索 (Dense + Sparse + RRF) | 🔥 **核心升级** |
| **向量模型** | MiniLM (384维) | bge-m3 (1024维) | 中英文语义更准 |
| **数据库** | ChromaDB (内嵌) | PostgreSQL + pgvector | 生产级可扩展 |
| **Sparse 检索** | ❌ 无 | ✅ tsvector/lexical | 关键词精确匹配 |
| **Scope 权重** | ❌ 无 | ✅ persistent > ephemeral | 长期记忆优先 |
| **过期管理** | 手动 | 自动 TTL + cleanup | 自动清理 |

## 2. 核心升级：混合检索 (Hybrid Search)

**这才是最大的提升，不是维度！**

### 旧系统检索流程

```
query → [向量相似度] → 结果
"苹果财报" 可能匹配到 "水果苹果" (语义相近但错误)
```

### 新系统检索流程

```
query → [向量相似度] + [关键词匹配] → RRF 融合 → 结果

"苹果财报"
  ├─ Dense: 语义相似的文档
  └─ Sparse: 必须包含"苹果"和"财报"关键词
→ 两者融合，精准度大幅提升
```

### RRF 融合公式

```python
# backend/rag/hybrid_service.py 第 514-520 行
rrf_score = 1/(k + dense_rank) + 1/(k + sparse_rank) + scope_boost
```

## 3. 为什么混合比纯向量好？

| 场景 | 纯向量 | 混合 |
|------|--------|------|
| "AAPL 2024 Q3 财报" | 可能返回 TSLA 财报（语义相似） | 必须包含 AAPL + Q3 关键词 |
| "什么是市盈率" | 精准 | 同样精准 |
| 专业术语 / 代码 | 容易语义漂移 | 关键词兜底 |

## 4. 维度高 = 厉害？

**不一定！** (￣^￣)

| 维度 | 影响 |
|------|------|
| 更高维度 | 表达能力更强，但计算成本更高 |
| 384 vs 1024 | 对于简单场景差异不大 |
| **真正差异** | 模型训练数据和目标才是关键 |

### bge-m3 厉害在于：

1. **多语言优化** — 中英文都强
2. **同时输出 Dense + Sparse** — 一次推理两种向量
3. **针对检索任务训练** — 不是通用 embedding

## 5. 旧系统短板

| 问题 | 旧系统表现 |
|------|-----------|
| 金融术语检索 | 语义漂移严重（"PE" 可能匹配到"体育"） |
| 中英文混合 | MiniLM 对中文支持一般 |
| 生产部署 | ChromaDB 单机，无法水平扩展 |
| 长期记忆优先 | 无 scope 权重机制 |

## 6. 升级重要性总结

| 升级项 | 重要性 | 说明 |
|--------|--------|------|
| Dense + Sparse 混合 | ⭐⭐⭐⭐⭐ | **最关键升级，精准度质变** |
| PostgreSQL + pgvector | ⭐⭐⭐⭐ | 生产级，可扩展 |
| bge-m3 模型 | ⭐⭐⭐ | 中英文更好 |
| 维度 384→1024 | ⭐⭐ | 锦上添花，不是核心 |

> **一句话**：升级的核心是「混合检索」，不是「维度更高」
