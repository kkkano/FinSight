## 1. 现状：我们缺什么评估

### 已有（检索质量层）

`tests/retrieval_eval/` 已经覆盖了**检索层**的质量评估：

| 指标 | 含义 | 当前基线（2026-02-07） |
|------|------|----------------------|
| Recall@6 | 相关文档被找到的比例 | 1.000 ✓ |
| Precision@6 | 找到的文档中相关的比例 | 0.400 ✓ |
| MRR | 第一条相关文档的排名 | 1.000 ✓ |
| nDCG@6 | 考虑排序质量的综合检索指标 | 1.000 ✓ |
| Citation Coverage | 引用覆盖率 | 1.000 ✓ |
| Latency P95 | 第95百分位延迟 | 0.08ms ✓ |

### 缺失（生成质量层）❌

`tests/rag_quality/`（本次新建）填补的空白：

| 指标 | 含义 | 之前状态 |
|------|------|---------|
| **Faithfulness** | 答案是否有文档依据（防幻觉） | ❌ 完全没有 |
| **Answer Relevancy** | 答案是否回答了问题 | ❌ 完全没有 |
| **Context Precision** | 检索文档的利用率 | ❌ 完全没有 |
| **Context Recall** | 关键信息的覆盖率 | ❌ 完全没有 |

### 一句话总结

> **以前只知道"找到了什么"，不知道"说对了什么"。**
> 就像考试只检查了"翻书翻到没有"，没检查"答案写对了没有"。

## 2. 为什么需要评估

### 金融场景的特殊风险

普通聊天机器人说错了，用户顶多觉得没用。
**金融分析系统说错了，用户可能做出错误的投资决策。**

```
用户问：茅台 Q3 营收多少？
LLM 幻觉答：约 450 亿元，增长 25%
真实数据：387.88 亿元，增长 15.6%

差额：62 亿元，增速偏差 60%——这不是"小问题"
```

### 三个核心问题

| 问题 | 对应指标 | 后果（如果不检测） |
|------|----------|-------------------|
| 答案里有没有凭空捏造的数据？ | Faithfulness | 用户看到错误的财务数据 |
| 答案有没有回答用户的问题？ | Answer Relevancy | 答非所问，用户体验极差 |
| 检索到的文档有没有被有效利用？ | Context Precision / Recall | 明明找到了正确文档，却没用上 |

### 没有评估体系会怎样

- **Prompt 改了**：不知道改好了还是改坏了
- **Chunker 参数调整**：不知道对生成质量有没有影响
- **新增文档类型**：不知道新类型的检索+生成效果
- **模型版本升级**：不知道新模型是否更容易幻觉

有了评估体系，每次变更都有数据支撑，不再靠"感觉"。

```

    │
    ▼

│  检索层（HybridRAGService）                               │
│  ┌─────────────────────────────────────────────────────┐ │

│  └─────────────────────────────────────────────────────┘ │

│    Recall@K / Precision@K / MRR / nDCG / Latency         │
└──────────────────────────────────────────────────────────┘
    │ 返回 Top-K 文档（上下文）
    ▼
┌──────────────────────────────────────────────────────────┐
│  生成层（LLM）                                            │
│  ┌─────────────────────────────────────────────────────┐ │
│  │ 上下文 + 问题 → LLM → 答案（报告/对话回复）         │ │
│  └─────────────────────────────────────────────────────┘ │

│    Context Precision / Context Recall                     │
└──────────────────────────────────────────────────────────┘
    │
    ▼
最终输出（报告 / 对话答案）
```

## 4. 一条 Query 的完整评估流程

### 示例 Query

> 用户输入：**"贵州茅台 2024 年三季度营业收入是多少？"**

---

### Step 1：检索（retrieval_eval 已覆盖）

```python
results = hybrid_service.retrieve("贵州茅台2024年三季度营业收入", top_k=6)
```

返回 6 条文档，期望排第一的是季报原文：

```
[文档1] 贵州茅台股份有限公司2024年第三季度报告：公司实现营业收入387.88亿元，
        较上年同期增长15.6%。其中茅台酒实现收入327.41亿元...
[文档2] 三季度公司实现净利润195.54亿元，同比增长14.4%...
[文档3] 白酒行业2024Q3综述：头部品牌均实现稳定增长...（相关但非直接答案）
[文档4-6] 其他背景信息...
```

**retrieval_eval 在这里检测**：文档1是否被召回？排第几？

---

### Step 2：生成答案

LLM 读取上述 6 条上下文，生成：

```
答案 A（高质量）：
"贵州茅台2024年三季度营业收入为387.88亿元，同比增长15.6%。
 其中茅台酒收入327.41亿元，系列酒收入52.24亿元。"

答案 B（幻觉）：
"贵州茅台2024年三季度营业收入约为450亿元，同比增长约25%，
 创历史新高，超出市场预期。"

答案 C（答非所问）：
"贵州茅台是中国最大的白酒生产企业，以其高端品牌定位著称，
 53度飞天茅台是其核心产品..."
```

---

### Step 3：四维 RAGAS 评估

#### 3.1 Faithfulness（忠实度）评估答案 A vs B vs C

RAGAS 用 LLM 将答案拆解为多个「原子陈述」，逐一验证是否有上下文支撑：

**答案 A 的原子陈述：**
```
陈述①：营业收入为387.88亿元     → 文档1有明确记载 ✓
陈述②：同比增长15.6%            → 文档1有明确记载 ✓
陈述③：茅台酒收入327.41亿元     → 文档1有明确记载 ✓
陈述④：系列酒收入52.24亿元      → 文档1有明确记载 ✓

Faithfulness = 4/4 = 1.00 ✓ 满分
```

**答案 B 的原子陈述：**
```
陈述①：营业收入约450亿元        → 文档中无此数据 ✗（幻觉！）
陈述②：同比增长约25%            → 文档中无此数据 ✗（幻觉！）
陈述③：创历史新高               → 文档中无此表述 ✗（幻觉！）
陈述④：超出市场预期             → 文档中无此表述 ✗（幻觉！）

Faithfulness = 0/4 = 0.00 ✗ 灾难性失败
```

**答案 C 的原子陈述：**
```
陈述①：最大白酒生产企业         → 文档中未提及 ✗
陈述②：53度飞天是核心产品       → 文档中未提及 ✗

Faithfulness = 0/2 = 0.00 ✗ 虽非幻觉，但完全不靠谱
```

---

#### 3.2 Answer Relevancy（答案相关性）

RAGAS 从答案逆向生成多个问题，计算与原始问题的语义相似度：

**答案 A 的逆向问题：**
```
生成问题①："贵州茅台Q3营收是多少？"  → 相似度 0.98 ✓
生成问题②："茅台酒和系列酒各贡献多少？" → 相似度 0.82 ✓

Answer Relevancy = 平均相似度 = 0.90 ✓
```

**答案 C 的逆向问题：**
```
生成问题①："贵州茅台是什么公司？"    → 相似度 0.32 ✗（原问题问的是营收）
生成问题②："茅台的核心产品是什么？"   → 相似度 0.28 ✗

Answer Relevancy = 0.30 ✗ 答非所问
```

---

#### 3.3 Context Precision（上下文精确率）

检索到的 6 条文档中，被答案实际利用的比例（靠前的文档权重更高）：

```
[文档1] 季报营收数据  → 答案直接引用   → 相关 ✓  排第1，权重最高
[文档2] 净利润数据    → 答案引用了部分  → 相关 ✓  排第2
[文档3] 行业综述      → 答案未使用     → 不相关 ✗ 排第3（影响精确率）
[文档4] 竞品信息      → 答案未使用     → 不相关 ✗
[文档5] 历史价格      → 答案未使用     → 不相关 ✗
[文档6] 宏观数据      → 答案未使用     → 不相关 ✗

Context Precision ≈ 0.58（2个相关文档在6个中，且考虑排序权重）
→ 说明检索引入了较多噪声，可优化 reranker 或 top_k 参数
```

---

#### 3.4 Context Recall（上下文召回率）

将 ground_truth 拆解为多个陈述，检查每条是否在上下文中有支撑：

```
ground_truth = "387.88亿元，同比增长15.6%，茅台酒收入327.41亿元，系列酒52.24亿元"

陈述①：387.88亿元     → 文档1有记载 ✓
陈述②：增长15.6%      → 文档1有记载 ✓
陈述③：茅台酒327.41亿 → 文档1有记载 ✓
陈述④：系列酒52.24亿  → 文档1有记载 ✓

Context Recall = 4/4 = 1.00 ✓
→ 检索覆盖了所有关键信息，检索质量极佳
```

---

### 最终评估结果汇总

| 答案 | Faithfulness | Answer Relevancy | Context Precision | Context Recall | 综合评价 |
|------|:---:|:---:|:---:|:---:|------|
| 答案 A（高质量） | 1.00 ✓ | 0.90 ✓ | 0.58 ⚠ | 1.00 ✓ | 答案好，检索有噪声 |
| 答案 B（幻觉） | 0.00 ✗ | 0.85 ✓ | 0.58 ⚠ | 1.00 ✓ | 致命幻觉，不可用 |
| 答案 C（偏题） | 0.00 ✗ | 0.30 ✗ | 0.58 ⚠ | 1.00 ✓ | 完全失效 |

> ⚠ Context Precision 偏低（0.58）说明检索层引入了太多无关文档，
> 需要调整 `reranker.py` 的截断策略或降低 `top_k`。

## 5. 四项核心指标详解

### 5.1 Faithfulness（忠实度）

**一句话**：答案说的每一句话，都能在检索文档里找到出处。

**计算方式**：
```
1. 用 LLM 把答案拆成 N 个原子陈述
2. 对每个陈述，用 LLM 判断是否有上下文支撑
3. Faithfulness = 有依据的陈述数 / 总陈述数
```

**分数解读**：
```
1.00      完美：每句话都有出处
0.80~0.99 良好：偶有推论性表述，主要内容可溯源  ← 金融场景最低要求
0.60~0.79 警告：约 1/4 内容可能是幻觉
< 0.60    危险：超过 40% 内容无依据，不可信任
```

**金融场景为何要求最严（min=0.80，filing 要求 min=0.85）**：
- 季报数字差 0.1% 就是重大误差
- 投资者依赖这些数据做决策
- 幻觉 = 潜在的法律风险

---

### 5.2 Answer Relevancy（答案相关性）

**一句话**：答案有没有真正回答用户的问题。

**计算方式**：
```
1. 用 LLM 从答案逆向生成 3~5 个问题
2. 计算这些生成问题与原始问题的 Embedding 余弦相似度
3. Answer Relevancy = 平均相似度
```

**分数解读**：
```
> 0.88    优秀：答案直接、精准地回答了问题
0.75~0.88 合格：答案有一定背景信息，核心问题被回答  ← 最低要求
0.60~0.75 偏低：答案有些跑题，背景信息过多
< 0.60    失败：答非所问
```

**典型失败场景**：
- 用户问"营收多少"，答案重点介绍公司背景
- 用户问"风险有哪些"，答案只说"本公司经营稳健"
- 用户问具体数字，答案给出趋势描述

---

### 5.3 Context Precision（上下文精确率）

**一句话**：检索回来的文档，有多少是真的被用到的（且越靠前越好）。

**计算方式**（Ranking-Aware）：
```
按位置加权：第1条文档权重最高，第6条最低
Precision@k = 前k条中相关文档数 / k
Context Precision = 加权平均
```

**分数解读**：
```
> 0.85    优秀：检索精准，几乎无噪声
0.70~0.85 合格：少量无关文档，不影响生成质量  ← 最低要求
0.50~0.70 偏低：检索引入明显噪声，LLM 需要过滤
< 0.50    危险：一半以上是无关文档，可能误导 LLM
```

**低精确率的常见原因**：
- `top_k` 设置过大（6→8），引入过多边际文档
- Reranker 效果不佳，相关文档未排到前面
- Embedding 模型对特定领域术语理解不足

---

### 5.4 Context Recall（上下文召回率）

**一句话**：标准答案所需的所有信息，有没有被检索回来。

**计算方式**：
```
1. 用 LLM 把 ground_truth 拆成 N 个陈述
2. 对每个陈述，判断是否能在检索文档中找到依据
3. Context Recall = 有依据的陈述数 / 总陈述数
```

**分数解读**：
```
> 0.85    优秀：所有关键信息都被检索到
0.70~0.85 合格：偶有细节遗漏，核心信息覆盖  ← 最低要求
0.50~0.70 偏低：重要信息有遗漏，答案会不完整
< 0.50    危险：超过一半关键信息没被检索到
```

**低召回率的常见原因**：
- `top_k` 设置过小（需要的信息在第7条但只取6条）
- Chunker 把关键数据截断在不同片段
- 文档没有入库（新上传的文档未被索引）

## 6. 阈值设定与参考标准

### 项目阈值（`tests/rag_quality/thresholds.json`）

| 指标 | 最低合格 | 优秀线 | Filing 专项 | 设定依据 |
|------|:-------:|:------:|:----------:|---------|
| Faithfulness | **0.80** | 0.90 | **0.85** | 金融数据零幻觉容忍；filing 更严因为数字精确性要求最高 |
| Answer Relevancy | **0.75** | 0.88 | 0.75 | 允许少量背景信息补充，但核心问题必须被回答 |
| Context Precision | **0.70** | 0.85 | 0.70 | 允许少量噪声文档，不影响主要生成质量 |
| Context Recall | **0.70** | 0.85 | **0.75** | Filing 关键财务数据必须被完整检索 |

### 行业参考基准

| 场景 | Faithfulness | Answer Relevancy | 来源 |
|------|:---:|:---:|------|
| 通用问答 RAG | 0.75 | 0.70 | RAGAS 官方基准 |
| 企业知识库 | 0.80 | 0.75 | Hugging Face RAG 评测 |
| **金融分析（本项目）** | **0.80+** | **0.75+** | 本项目定制 |
| 医疗诊断辅助 | 0.90+ | 0.85+ | 高风险领域要求 |

> 金融场景介于「企业知识库」和「医疗辅助」之间，阈值合理。

### 漂移容忍设定

```
每项指标允许最大退步：-0.05（即 5 个百分点）

理由：
- 模型随机性会带来 ±1~2% 的正常波动
- 允许 5% 是为了容纳正常波动，超过则说明有实质变化
- 比 retrieval_eval 的漂移容忍（-0.03）更宽松，
  因为生成质量天然有更大的方差
```

## 7. 如何运行评估

### 环境准备

```bash
# 安装 RAGAS 依赖
pip install "ragas>=0.2.0"

# 设置 LLM API（使用与生产不同的模型以保证客观性）
export LLM_API_KEY="your-api-key"
export LLM_API_BASE="https://api.openai.com/v1"  # 或其他兼容接口
export EVAL_LLM_MODEL="gpt-4o-mini"               # 推荐：便宜+够用
```

### 日常使用

```bash
# 1. 首次运行：完整评估 + 保存基线
python tests/rag_quality/run_rag_quality.py --save-baseline

# 2. 常规评估（对比基线，检查有无退步）
python tests/rag_quality/run_rag_quality.py

# 3. CI 门控模式（失败则退出码 1，阻断 pipeline）
python tests/rag_quality/run_rag_quality.py --gate

# 4. 只评估季报类文档（更严格的 filing 阈值）
python tests/rag_quality/run_rag_quality.py --doc-type filing --gate

# 5. Mock 模式（无需真实 LLM，验证框架本身）
python tests/rag_quality/run_rag_quality.py --mock

# 6. 查看帮助
python tests/rag_quality/run_rag_quality.py --help
```

### 快速单元测试（无需 LLM）

```bash
# 只跑 rag_quality 的 pytest 快速测试（< 500ms）
pytest tests/rag_quality/ -v

# 跑所有测试（含 retrieval_eval）
pytest tests/ -v --ignore=tests/regression

# 典型输出：
# tests/rag_quality/test_rag_quality.py::TestDatasetStructure::test_dataset_file_exists PASSED
# tests/rag_quality/test_rag_quality.py::TestThresholdsConfig::test_min_values_in_valid_range PASSED
# tests/rag_quality/test_rag_quality.py::TestGateCheckerLogic::test_passing_metrics_no_failures PASSED
# ... 共 18 个测试
```

### CI 集成建议

```yaml
# .github/workflows/quality.yml（示例）
- name: 检索质量门控（快速，< 30s）
  run: python tests/retrieval_eval/run_retrieval_eval.py --gate

- name: RAG 生成质量门控（需要 LLM API，约 3~5 分钟）
  env:
    LLM_API_KEY: ${{ secrets.EVAL_LLM_API_KEY }}
  run: python tests/rag_quality/run_rag_quality.py --gate
  # 建议：仅在 main 分支合并时运行，PR 阶段只跑快速 pytest
```

## 8. 与 retrieval_eval 的关系

```
两套评估互补，缺一不可：

retrieval_eval                    rag_quality（本文档）
─────────────────────────         ────────────────────────────────
检测：找到了正确文档吗？          检测：用文档生成了正确答案吗？
层级：向量检索层                  层级：LLM 生成层
速度：< 100ms（无 LLM）           速度：3~5 分钟（需要 LLM）
频率：每次 PR 都跑                频率：合并到 main 时跑
失败原因：                        失败原因：
  - Embedding 模型退化              - LLM 幻觉增加
  - Chunker 截断关键信息            - Prompt 模板劣化
  - 向量数据库索引损坏              - 上下文长度超限
  - RRF 参数配置错误                - 模型版本更换
```

**经典诊断场景**：

| retrieval_eval | rag_quality | 诊断结论 |
|:---:|:---:|------|
| ✓ 通过 | ✓ 通过 | 系统健康 |
| ✓ 通过 | ✗ 失败（低 Faithfulness）| 文档找对了，但 LLM 在幻觉——检查 Prompt |
| ✓ 通过 | ✗ 失败（低 Context Precision）| 检索排序不好，无关文档排太靠前——调 Reranker |
| ✗ 失败 | ✗ 失败（低 Context Recall）| 文档没被找到，生成层无米可炊——先修检索 |
| ✗ 失败 | ✓ 通过 | 罕见：可能恰好用了缓存结果，建议清缓存重跑 |

## 9. 常见问题 FAQ

**Q：为什么用 `gpt-4o-mini` 而不是 `gpt-4o` 做评估？**

A：评估需要大量 LLM 调用（12个案例 × 4个指标 × 多次内部推理），用 `gpt-4o` 成本过高。`gpt-4o-mini` 在判断文本蕴含关系（是否有依据）上足够准确，性价比最高。如果条件允许，可以专门对 `filing` 类案例使用 `gpt-4o`。

---

**Q：为什么不直接用 LangSmith 评估？**

A：可以用，但 LangSmith 是第三方服务，存在以下限制：
1. 金融数据敏感，数据出境合规问题
2. 每月有免费额度限制，超出收费
3. RAGAS 专为 RAG 设计，指标更精准（LangSmith 是通用评估平台）

本方案完全本地运行，数据不离开内网。

---

**Q：Faithfulness 分数低（< 0.80），该怎么优化？**

A：按优先级排查：
1. 检查 Prompt 是否强调"只使用文档内容"
2. 检查 `top_k` 是否过小导致关键信息没进入上下文
3. 检查是否有文档未入库（新增文档但未重建索引）
4. 考虑降低生成温度（`temperature=0`）
5. 对于 filing 类，确认表格被完整保留（`chunker.py` 表格保护逻辑）

---

**Q：Context Precision 一直偏低（0.55 左右），正常吗？**

A：如果 `top_k=6` 且只有 2 条文档真正相关，Precision 约为 0.5~0.6 是正常的。
优化方向：
- 将 `top_k` 从 6 降到 4（减少噪声）
- 优化 Reranker 截断（确保相关文档在前3）
- 但注意：降低 `top_k` 可能影响 Context Recall（覆盖率下降）
- 在 Precision 和 Recall 之间需要权衡，建议以 `faithfulness` 为最终锚点

---

**Q：如何扩充黄金数据集？**

A：建议每个季度增加 3~5 条新案例，特别是：
1. 近期出现过幻觉的真实 query（从用户反馈中收集）
2. 新增文档类型的代表性问题
3. 边界情况：数据缺失的问题、需要跨文档推理的问题

可以用 LLM 自动生成候选案例（基于现有文档），然后人工审核 `ground_truth`。
