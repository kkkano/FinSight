# FinSight AI 投研工作台 - 深度面试核心 Q&A 宝典

这份文档整理了关于 FinSight AI 底层架构（LangGraph + RAG + Multi-Agent）的最核心、最尖锐的面试问题。这不仅是面试武器，更是整个系统技术深度的最佳证明。

---

## 0. 破冰与认知：关于“职责边界模糊”的探讨

**Q：在我之前的技术评审中，曾有观点认为你的系统存在“各个 Agent 职责边界模糊”的问题。你认可这个观点吗？如果让你反驳，你会怎么证明你们边界是清晰的？**

**回答思路：**
表面上看，这确实是一个痛点，因为在复杂的投资研报生成中，几乎所有逻辑都在交叉（例如在分析公司基本面时不可避免要看新闻，看新闻也一定涉及股价波动）。如果完全解耦，最终的报告就会是几块孤立数据的拼接。但实际上，FinSight 的边界是非常克制和明确的：

1. **领域的水平切割（按数据源分界）：**
   - 我们严格定义了代理之间的边界：`news_agent` 只处理外部事件与情绪（它不能做财务测算）；`fundamental_agent` 核心基于财报和盈利预测（它不看K线）；`technical_agent` 专注量价指标（MA/RSI/MACD）。也就是“**每个人只能基于它看到的那一面信息产生洞察**”。
2. **逻辑的垂直切割（Planner vs Executing）：**
   - 控制流和数据流是分离的。`planner` 负责“我要查什么、顺序怎样”，它**严禁**生成任何最终分析；所有的分析必须在 `execute_plan` 中由各个 Agent 具体执行，最后由 `synthesize` 节点收口。
3. **消除“上帝视角代理”：**
   - 为了打破“边界模糊”，我们设计了 `Risk Agent` 和后期融合的 `Report Builder`。比如，即使新闻 Agent 说“情绪大好”，只要 K线技术 Agent 报了“超买顶背离”，最后是由整体框架（而非单个 Agent 擅自越权）来决断权重。

所以，与其说是“边界模糊”，不如说是**系统在 Synthesis 阶段实现了高级别的并发信息交融**。

## 1. 架构总览：LangGraph 总链路与状态机

**Q：介绍一下你们基于 LangGraph 的总工作流（Pipeline）是什么样的？一次典型的请求包含了哪些核心节点？**

**回答思路：**
我们的 LangGraph 编排（在 `backend/graph/runner.py`）是一个包含了 **18个节点** 的复杂有向无环图，但主线非常清晰。一次典型的投资分析汇报生成，链路如下：

1. **入口预处理：** `build_initial_state` 初始化状态 → `trim_history` 裁剪上下文保证不爆Token → `normalize_ui_context` 合并前端UI状态。
2. **意图路由段：** `decide_output_mode` 决定是闲聊还是生成研报 → 进入 `resolve_subject`（确定标的，如新闻还是个股）。如果是明确指令，直接进 `parse_operation`；如果需要澄清问题，会通过 `clarify` 后再往下走。
3. **规划门控段：** `policy_gate`（策略门控）决定本轮能用哪些工具和 Agent，防范越权 → `planner`（LLM 或者 Stub）拆解并生成执行计划（PlanIR，包含多步分组执行）。
4. **人工断点（Human-in-the-loop）：** 报告生成前，强制进入 `confirmation_gate`（人工确认门）。这会向外抛出中断（`interrupt()`），等待用户点击“确认执行”、“调整参数”或“取消”。
5. **执行与发版：** 确认后，进入 `execute_plan`，根据并行分组（parallel_group）并发调度多 Agent 或直接调工具 → 然后进入最重要的 `synthesize` 流式合成报告，最后通过 `render` 打包为前端可读的卡片/研报输出。

**Q：你们的状态机数据结构（State Machine）里维护了哪些信息？为什么要这么设计？**

**回答思路：**
LangGraph 的核心是 State 传递。我们的 `GraphState` (在 `state.py`) 继承自原生的 `MessagesState`，除了持有对话历史外，我注入了大量结构化信息字段：
- **核心查询状态**：`query`，`ui_context`。
- **业务上下文**：`subject` (包含主体类型 subject_type，如 company, filing, portfolio)，`operation` (操作名称及置信度)。
- **限制与预算**：`policy` (资源预算 budget，允许使用的 allowed_tools 和 allowed_agents，最高并发不超过 4 个 Agent)。
- **执行过程状态**：`plan_ir` (中间步骤表示)，`artifacts` (执行结果、证据池 evidence_pool、各个 Agent output 集合、融合后的最终 report)。
- **门控与追溯**：`confirmation_*` 系列字段处理断点恢复的意图，`trace` 记录耗时与容错。

这样设计的目的，是保证**系统任何时候中断、崩溃或进行人工审核打断（Interrupt）时，都能做 Postgres 检查点（Checkpoint）持久化，并且随时完美 Resume（恢复）**。

## 2. 深度检索机制：Hybrid RAG & 分块

**Q：为什么你们的 RAG 不直接用 Embedding 做密集检索，而是要用 BGE-M3 做 Sparse (BM25词频) + Dense 的混合检索？最后怎么融合？**

**回答思路：**
在金融投研业务中，仅靠 Dense 向量检索有着致命缺陷：它能理解“苹果公司的盈利能力”，但很容易漏掉例如编号“Q3-2025 EPS 1.34”、“A股代码 600519”这种**极度依赖精确匹配**的专有名词或编号。
1. **模型选型**：我们选用了 `BAAI/bge-m3`，因为它单次推理就可以同时产出 1024 维的 Dense Tensor，和包含词法权重的 Sparse Vector。
2. **融合方式**：我们在 `backend/rag/hybrid_service.py` 内部实现了 **RRF (Reciprocal Rank Fusion) 倒数排名融合算法**。我们不仅做了融合，还独创了**Scope Boost (范围提权)**。例如，持久化类型（Persistent，如企业年报）天然获得 `+0.15` 的提权，临时缓存（如几个小时内的新闻）没有提权。这也体现了金融研报中“第一信源优于二手新闻”的原则。
3. **最终过滤**：融合大名单后，再输入给 `bge-reranker-v2-m3` 交叉编码器进行精确打分和重排，最后送入大模型。

**Q：文档分块 (Chunking) 有什么特殊处理？这体现了你怎样的工程思考？**

**回答思路：**
我在 `backend/rag/chunker.py` 摒弃了呆板的 “统一 1000 字符切割”，而是做了高度垂直定制：
- **年报/Filing：** 采取“大块度有重叠”切割（1000组/200重叠）。
- **财报电话会 Transcript：** 采用**Q&A-aware感知切割**（800/100重叠）。我们会利用正则表达式或者分隔符把 `\nQ:` / `\nA:` 找出来，尽量保证一个分析师提问和管理层回答被框在一个 Chunk 中，防止上下文撕裂。
- **快讯/News：** 对于 2000 字以内的短消息，**完全不切割**，以保证事件叙事的完整性。
- **财务表格 Table：** 绝对不切割。切碎的表格是大模型的噩梦。这也是我们能做精准财报分析的关键之一。

## 3. 多智能体协作：Agent 范式与 Planner 工具调用

**Q：这可能是整个框架最核心的问题：按理说大模型规划器（Planner）应该只负责调度 Agent（比如委派 News Agent 去查新闻），但是在你们系统中，Planner 会直接生成包含“工具（Tool）”的调用步骤。让 Planner 自己调工具、越过 Agent，是什么考量？**

**回答思路：**
这是一个极其深入且针对性的好问题！
如果一味遵循纯原教旨主义的 Multi-Agent，所有活都应该抛给 Agent。但我们在真实投研业务中发现，**“层层发包”会导致严重的延迟和昂贵的 Token 开销**。
在 `planner.py` / `planner_stub.py` 及 `execute_plan_stub.py` 底层，我们的思路是**快慢分离（Fast & Slow Path）**：
1. **轻量级数据获取（直接 Tool 获取）：** 比如提取实时的公司股价，拿公司简介。这不需要任何推理论述，只需要拿到 JSON 数据组装到证据池即可。给 Planner 直接指派 `kind="tool"` 并且并行执行的权利，可以省掉封装 `PriceAgent` 去发散反思（Reflection）的时间。（在预算预估中，Tool 只占 0.6个开销单位）。
2. **重度逻辑推演（分发给 Agent 分析）：** 比如要汇总宏观情绪、生成具有观点性的多维评级时，Planner 就不能直调，必须调用 `kind="agent"`，利用其背后的 ReAct 或者 CoT 进行推演处理（宏观 Agent 占 2.6，深搜占 3.8个开销单位）。
3. 这个混编架构保证了系统既具备高度敏捷的数据拉取能力（类似外挂脚本），又通过专门的 Sub-Agent 保证了深度推演的上限。这恰恰是工程落地的精髓。

**Q：详细说说子代理（Sub-Agents）的设计，如何根据金融场景区分它们的 Reflection 反思范式？**

**回答思路：**
总共实现有 7 个代理，各自绑定了最适合该领域的推理模式：
1. **DeepSearch Agent (深入搜研)：** 使用 `Self-RAG` 模式。它有内置判别器（SearchConvergence），会多次反思“新搜到的财报信息对最初的问题有增量帮助吗（Info-gain scoring）”？如果是，就继续扒，最多循环两次。
2. **Fundamental Agent (基本面)：** 使用 `Chain-of-Thought (CoT)`。对利润表、资产负债表不需要试错，只需要严密的数学运算和一步步推证逻辑。
3. **News Agent (新闻)：** 使用经典的 `ReAct` 模式。看了新闻标题 → 发现有重大利空 → 思考是否需要继续挖细节 → 动用 `get_news_sentiment` 工具进一步查情感评分。
4. **Technical Agent (技术面)：** 使用 `Signal Confluence` 模式，将离散的 K线 MA均线、MACD指标拿进来做信号共振判定。
5. **Macro Agent (宏观)：** 使用 `Plan-Execute-Reflect`。
6. **Risk Agent (风控)：** 唯一**非大模型主导（纯规则）**的评估器。为什么这么设计？因为合规与风险评级是底线，不能容忍 LLM 任何随机性。它严格提取各子代理发出的冲突标志、下跌阈值等强数据，生成不可篡改的风险强警告。

## 4. Benchmark的灵魂拷问与质量防线

**Q：尖锐提问：“你提到在 Quality Engine 里对接地率(Grounding rate)和输出质量做了控制，但这些 Benchmark 本质都是你自己定义的评估规则，既没有行业公认打分也没有人工复核团队评估，这分数的参考价值在哪里？是不是属于虚高或者自嗨？”**

**回答思路（不卑不亢，展示工程逻辑）：**
这个问题非常犀利。如果我只是简单设了一个 `random.uniform(0.8, 1.0)` 那确实是自嗨；但在我们的 `backend/report/quality_engine.py`中，我们的 Benchmark 是**“过程强约束型指标”，而不是“最终的主观审美指标”**。
它有参考价值的原因如下：
1. **指标是客观可算计的（Grounding/Citations）：** 我们的 `Grounding rate` 并不是让 LLM 自己说“我写得好不好”，而是严格追踪在最终生成的 Report 里面，**引用的角标（Citations）与系统供给的证据池（Evidence Pool）对应占比**。如果占比低（例如低于 60%），说明输出内容中出现大量没出处的幻觉信息，在系统里直接触发 `QUALITY_PROFILE_CRITICAL_MISSING`，评定为 `block`，不仅阻止本次上报，还会阻断长尾缓存索引。
2. **硬性的校验拦截门限（Hard Gates）：** 相比于在测试集上刷个高分，我们的阈值是在 Runtime（运行时）直接作用的。我们规定 `warn` 级别（例如引用率偏低但还能看）打 Tag 放行，但 `block` 级别（例如多次搜不到信息却生造结论、或者触发合规屏蔽）是断头台逻辑，拦截发表并报错。
3. **总结：** 这个框架基准的设计目的不是去竞争大模型能力的跑分排行榜，而是**构建一个工程级别防止投资建议发生致命灾难的防线底座**。我们关注的是“不可反驳的异常阻断”，这个客观拦截机制是扎扎实实保障投研报告一致性的，绝不是简单的粉饰。

## 5. 补充加分项：容错与执行

**Q：讲一个小亮点，你在处理大语言模型返回的规划（PlanIR / JSON）时，是否有被 JSON 解析失败坑过？怎么解决的？**

**回答思路：**
LLM 输出 JSON 是极其容易崩溃的坑，尤其是偶尔混入未转义的双引号、或者多出尾排逗号（Trailing comma）、或者甚至外层少一个 `}` 等。这也是 Agent 系统最容易卡死的原因。
在 `planner.py` 里，我手写了一套强悍的兜底清洗 Pipeline (`_repair_json_text`):
1. **基础清洗：** 用正则去掉类似 Markdown ````json ... ```` 的包裹。
2. **控制字符过滤：** `_sanitize_control_chars`，把由于粘包带过来的特殊占位符去掉。
3. **修复双引号：** 处理由于智能引号或嵌套 `"` 导致的 JSON 格式损坏。
4. **尾部修复与兜底转换：** 甚至加入了 `ast.literal_eval` 作为 Python 安全对象的终极 fallback 处理。
这些小细节使得 Planner 即便面临模型幻觉抽风，依然保证系统健壮性，实现了从一次请求到返回的 100% 连贯跑通。
